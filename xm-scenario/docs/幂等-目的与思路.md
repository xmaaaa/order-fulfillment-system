# 幂等的目的与思路

## 一、幂等的目的

**一句话**：在「同一次业务意图被多次触发」时，只产生一次效果，且多次调用得到的结果一致。

- 典型场景：重复点击、网络重试、消息重复投递、下游重复调用。
- 不做幂等的后果：重复扣款、重复下单、重复发货、数据被错误覆盖。

所以目的就是：**一次业务意图 ↔ 一次效果；多次触发 ↔ 结果一致**。

---

## 二、幂等的本质：两件事

幂等要解决两件事：

1. **标识**：能唯一标识「这一笔业务意图」或「这一份资源当前状态」。
2. **决策**：根据这个标识，决定是「执行一次并记结果」还是「直接返回已有结果 / 不修改」。

不同做法只是「用什么做标识」和「用什么机制做决策」的差别。

---

## 三、新增、更新、删除的幂等是一套思路吗？

是。**都是「标识 + 决策」**，只是标识和决策的具体形式不同：

| 操作 | 标识（认的是谁） | 决策（怎么保证只一次/结果一致） |
|------|------------------|----------------------------------|
| **新增** | 请求维：幂等键（如 clientId:requestId）；或资源维：业务唯一键（如订单号） | 请求维：第一次执行并存结果，同 key 直接返回缓存；资源维：唯一键已存在则返回「已存在」不重复建。 |
| **更新** | 资源 ID + 版本/状态（或再加 requestId） | 资源维：乐观锁 / 状态机 CAS，只有「当前版本/状态符合预期」才改；重复请求会「条件不满足、不更新」，结果一致。也可用请求维：同一 requestId 的更新只执行一次并缓存。 |
| **删除** | 资源 ID（必要时 + 版本/状态） | 第一次删除成功；后续同 ID 再删 → 已删就返回「已删除」或 0 行，结果一致。软删用状态机「已删除」不再变。 |

所以：**不是「只有新增才需要幂等」**，更新、删除也都要；只是**新增常用「幂等键 + 存结果」**，**更新/删除常用「资源 ID + 乐观锁/状态机」做 CAS**。本质都是：**认准「这一笔」→ 只执行一次或改不动/状态不变**。

---

## 四、两种实现思路（可以同时用）

可以归纳成两条互补的线：

### 1. 请求维度幂等（认「这次请求」）

- **标识**：客户端或网关给的**幂等键**（requestId、token、业务单号+操作类型等）。
- **决策**：  
  - 先查存储里有没有这个 key；  
  - 没有 → 执行逻辑，把结果存下来，返回；  
  - 有 → 不执行，直接返回已存的结果。

适合：**新增、或任意「同一次请求只允许执行一次」的写操作**。  
你项目里的 `IdempotentOrderCommandService` 就是这种：用幂等键 + 首次结果缓存，只对「创建草稿」做了请求维度幂等。

### 2. 资源维度幂等（认「这个资源当前状态」）——本质就是 CAS

- **标识**：资源的**版本号（version）**或**状态（state）**。
- **决策**：  
  - 只有「当前版本/状态等于我期望的」才更新；  
  - 否则不更新（或返回冲突），重复请求会「条件不满足」，结果一致。

这就是你说的：**乐观锁、状态机本质上都是一种 CAS**——都是「先比再改」：

- **乐观锁**：Compare = 库里的 version = 我读到的旧 version；Swap = 写新值且 version+1。重复请求 Compare 失败，不更新。
- **状态机**：Compare = 当前状态允许该事件；Swap = 状态流转。已在终态或不允许则 Compare 失败，重复请求状态不变。

所以：**更新、删除的幂等，常用「资源维度 CAS」**（乐观锁/状态机）；**新增常用「请求维度幂等键」**；两者可以叠加使用。

## 五、时效性 token/ID 过期瞬间的不一致

你的担心对：**如果只靠「带过期时间的 id/token」判断是否第一次，过期那一瞬间可能有两个请求都认为自己是「过期后第一个」**，就会都去执行，产生重复。

正确做法是：**不要把「唯一性」只押在时间上**，而要押在「存储或资源的约束」上：

1. **幂等键**：用**存储的唯一性**保证「同一 key 只成功一次」。例如幂等表里 `idempotency_key` 唯一索引，第一次 INSERT 成功，第二次同 key INSERT 会冲突，直接查已缓存结果返回；**TTL 只用来定时清理历史数据**，不参与「是否第一次」的判断。这样过期瞬间也不会出现两个都当成第一次。
2. **资源 CAS 兜底**：即使某次请求被误当成「新请求」执行了，更新时还有**乐观锁/状态机**：重复的那次会 version 不匹配或状态不允许，改不动，结果仍然一致。
3. **分布式锁**：对「幂等键 + 执行」加锁，同一 key 同一时刻只有一个在执行，也能避免过期瞬间竞态（通常配合存储唯一性用）。

结论：**幂等的「只执行一次」应依赖存储唯一约束或资源 CAS，而不是依赖「没过期」**；时间只负责失效与清理。

## 六、在本项目里的对应

| 思路 | 项目里的体现 |
|------|--------------|
| 请求维度幂等 | `IdempotentOrderCommandService`：幂等键 + `IdempotencyKeyStore` + 首次结果缓存；只对「创建草稿」做了示例，其它写操作也可以套同一套「幂等键 + 存结果」。 |
| 资源维度 CAS | `Order.version` + `OrderRepository.updateVersion(order)`（乐观锁）；`OrderStateMachine` + `Order.transition()`（状态机）；重复提交/重复消息会 CAS 失败或状态不变，结果一致。 |
| 新增 vs 更新/删除 | 新增：用幂等键；更新/删除：用乐观锁、状态机（或再加幂等键做请求维度）。 |

---

## 七、总结

- **目的**：一次业务意图只产生一次效果，多次触发结果一致。
- **本质**：**标识**（这一笔请求 or 这一份资源）+ **决策**（只执行一次 or CAS 不生效）。
- **请求维度**：幂等键 + 存结果，适合新增或「同请求只执行一次」。
- **资源维度**：乐观锁、状态机 = CAS，适合更新、删除；重复请求会「改不动」或「状态不变」。
- **时效 token**：TTL 只做清理；唯一性用存储唯一约束或资源 CAS 保证，避免过期瞬间不一致。
- **新增/更新/删除**：都可以幂等，只是标识和决策的具体形式不同；思路统一为「标识 + 决策」。

这样就能在「目的 + 思路」层面完全理解幂等，再落到具体实现（幂等键、乐观锁、状态机、token）就不会混了。
