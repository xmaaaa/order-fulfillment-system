# 状态机用到的设计思想与模式

## 一、整体结构对应

| 类/概念 | 设计思想 / 模式 | 说明 |
|--------|-----------------|------|
| **OrderState / OrderEvent** | **有限状态机 (FSM)** | 状态 = 节点，事件 = 边；合法流转由「当前状态 + 事件」唯一决定下一状态。 |
| **OrderStateMachine** | **表驱动状态机 (Table-Driven FSM)** | 用 `Map<State, Set<Event>>` + `next(state, event)` 表达流转表，结构清晰、易扩展（可改为从配置/DB 加载）。 |
| **TransitionGuard** | **策略模式 (Strategy) + 规格模式 (Specification)** | 见下节。 |
| **OrderStateMachineGuards** | **具体策略/规格 + 组合** | 多个 Guard 用 `&&` 组合，相当于「复合规格」。 |
| **Order.transition(event, guard)** | **模板方法 (Template Method) + 策略 (Strategy)** | 流程固定：先查 FSM → 再查 Guard → 再改状态；Guard 是可插拔的策略。 |

---

## 二、TransitionGuard / OrderStateMachineGuards 算什么设计模式？

### 1. 策略模式 (Strategy)

- **定义**：把「是否允许流转」这一**算法/策略**抽象成接口，不同实现可替换（无守卫、按金额、按行数、按地址等）。
- **对应**：
  - **Context**：`Order.transition(event, guard)` 的流程（用哪个 guard 由调用方传入）。
  - **Strategy 接口**：`TransitionGuard`（`boolean allow(state, event, order)`）。
  - **具体策略**：`PAY_AMOUNT_POSITIVE`、`SUBMIT_HAS_LINES`、`TransitionGuard.none()`。
- **效果**：不改 `Order`、不改 `OrderStateMachine`，只新增/替换 Guard 实现就能加规则（开闭原则）。

### 2. 规格模式 (Specification)

- **定义**：把**业务规则**封装成「谓词」（返回 true/false 的对象），可组合、可复用。
- **对应**：
  - **Specification 接口**：`TransitionGuard` 就是一个 Specification——问「(state, event, order) 是否满足某条业务规则」。
  - **具体规格**：`PAY_AMOUNT_POSITIVE` = 「支付时金额必须 > 0」；`SUBMIT_HAS_LINES` = 「提交时必须有行且总价非负」。
  - **组合**：`defaultGuards()` 用 `&&` 把多条规格组合成一条复合规格。
- **效果**：业务规则以「规格」形式独立存在，易读、易测、易扩展。

### 3. 小结：Guard 的双重身份

| 视角 | 模式 | 一句话 |
|------|------|--------|
| 行为可替换 | **Strategy** | 「是否允许流转」的策略可插拔，Order 依赖抽象 Guard。 |
| 规则即谓词 | **Specification** | Guard 是「满足/不满足」的规格，可组合（and/or）。 |

所以：**TransitionGuard / OrderStateMachineGuards 同时体现了策略模式 + 规格模式**；从「可替换策略」看是 Strategy，从「业务规则谓词 + 组合」看是 Specification。

## 三、其它相关思想

### 1. 模板方法 (Template Method) —— Order.transition

- **固定骨架**：① 用 FSM 算 next 状态；② 用 Guard 判是否允许；③ 允许则改 state 和 version。
- **可变点**：第 ② 步用哪个 Guard 由参数传入，相当于模板里的「钩子」。
- 所以整体是：**表驱动 FSM + 策略式 Guard + 模板方法式流程**。

### 2. 开闭原则 (OCP)

- 加新流转规则：在 `OrderStateMachine` 里加表项（或配置）。
- 加新业务条件：加新的 `TransitionGuard` 实现并组合进 `defaultGuards()`，不修改 `Order` 和 FSM 核心逻辑。

### 3. 单一职责 (SRR)

- **OrderStateMachine**：只负责「状态 + 事件 → 下一状态」。
- **TransitionGuard**：只负责「在 (state, event, order) 下是否允许」。
- **Order**：负责持有时序（state、version）和发起 transition，不写死具体规则。

## 四、和 GoF 的对应关系（简表）

| GoF 模式 | 在你状态机里的体现 |
|----------|--------------------|
| **State** | 用枚举 + 表驱动代替「每个状态一个类」，更轻量；本质仍是「状态决定行为」。 |
| **Strategy** | `TransitionGuard` 接口 + 多种实现（none、金额、行数、组合）。 |
| **Specification** | Guard 即规格：谓词 + 可组合（and）。 |
| **Template Method** | `Order.transition()` 的固定步骤 + Guard 作为可变步骤。 |

---

## 五、一句话总结

- **状态机本身**：**有限状态机 + 表驱动**（状态/事件/流转表）。
- **TransitionGuard / OrderStateMachineGuards**：**策略模式**（可插拔的「是否允许」策略）+ **规格模式**（业务规则谓词，可组合）。
- **Order.transition**：**模板方法**（固定流程）里注入 **Strategy**（Guard）。

这样就能把你状态机里用到的思想（FSM、表驱动、策略、规格、模板方法）和具体类一一对应起来。
